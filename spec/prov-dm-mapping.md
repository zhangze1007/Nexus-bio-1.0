# PROV-DM Mapping

## Scope

This document maps Nexus-Bio trust-runtime objects to PROV-DM concepts. It is an alignment guide, not a claim that the repository currently emits validator-backed PROV bundles.

## Mapping Summary

| Nexus-Bio trust-runtime object | PROV-DM concept | Meaning |
| --- | --- | --- |
| `Evidence` | `prov:Entity` | A source, dataset, simulation result, user input, experiment note, or manual review artifact used by a workflow. |
| `ProvenanceEntry` for a tool run | `prov:Activity` | A bounded run, import, export, human gate, or review that consumes and produces workflow artifacts. |
| Human reviewer or human gate actor | `prov:Agent` | A person or accountable system that performs or approves a review activity. |
| Output generated by a tool run | `prov:wasGeneratedBy` | The output entity should reference the tool-run activity that produced it. |
| Upstream provenance chain | `prov:wasDerivedFrom` | A downstream entity should reference upstream entities or provenance records it depends on. |
| Workflow bundle | `prov:Bundle` | A grouped set of entities, activities, agents, and relationships for one workflow or export package. |

## Object Semantics

### Evidence -> prov:Entity

`Evidence` is the closest Nexus object to `prov:Entity`. It should represent something that can be cited, reviewed, or inspected: a paper, dataset, user-supplied value, simulation output, experiment record, or manual review note.

Suggested PROV fields:
- `id`: `evidenceId`
- `prov:type`: `evidenceType`
- `prov:label`: `title`
- Nexus extension fields: `source`, `doi`, `url`, `notes`

### ProvenanceEntry / Tool Run -> prov:Activity

`ProvenanceEntry` represents a bounded activity. For a tool run, `activityType` is `tool-run`; for a human approval step, `activityType` is `human-gate` or `review`.

Suggested PROV fields:
- `id`: `provenanceId`
- `prov:type`: `activityType`
- `prov:startTime`: `startedAt`
- `prov:endTime`: `completedAt`
- Nexus extension fields: `toolId`, `inputAssumptionIds`, `outputAssumptionIds`

### Human Reviewer / Human Gate -> prov:Agent

When `actor` identifies a reviewer or accountable system, it aligns with `prov:Agent`. The protocol does not require storing personally identifying information; an internal reviewer ID or role can be enough.

Suggested PROV fields:
- `id`: reviewer or system ID
- `prov:type`: `Person`, `Organization`, or `SoftwareAgent` where known
- `prov:label`: human-readable actor label

### Output Generated By A Tool Run -> prov:wasGeneratedBy

A workflow output should link to the `ProvenanceEntry` activity that generated it.

Example relationship:

```json
{
  "prov:wasGeneratedBy": {
    "entity": "artifact:fbasim:flux-table:001",
    "activity": "provenance:fbasim-run:001"
  }
}
```

### Upstream Provenance Chain -> prov:wasDerivedFrom

`upstreamProvenanceIds` aligns with derivation. When a CETHX view consumes a PathD graph, or a DBTL export consumes multiple prior outputs, the derived artifact should preserve those upstream links.

Example relationship:

```json
{
  "prov:wasDerivedFrom": {
    "generatedEntity": "artifact:dbtlflow:handoff:001",
    "usedEntity": "artifact:pathd:pathway:001"
  }
}
```

### Workflow Bundle -> prov:Bundle

A complete Nexus-Bio workflow can be represented as a bundle containing:
- evidence entities,
- generated artifact entities,
- provenance activities,
- reviewer or tool agents,
- generation and derivation relationships.

This maps to `prov:Bundle` conceptually. A future implementation may serialize this as PROV-JSON or another PROV-compatible format after validation.

## Non-Compliance Statement

This repository does not currently claim full PROV-DM compliance. The mapping is intended to keep the trust-runtime protocol compatible with PROV-DM concepts while leaving validator-backed export, namespace policy, and complete serialization rules for future work.
